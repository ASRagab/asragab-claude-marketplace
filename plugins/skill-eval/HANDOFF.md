# skill-eval: Agent Handoff Document

> Written 2026-03-28, updated 2026-03-30. Covers the complete build-out of M1–M4, plugin packaging, and distribution cleanup.
> Intended audience: any agent (or human) picking up this project cold.

## What This Is

A 4-stage pipeline that mines Claude Code session transcripts for friction signals, identifies what to improve, and runs autoresearch-style optimization loops to generate and evaluate improvements automatically.

```
M1 (extract) → M2 (classify) → M3 (target) → M4 (autoresearch)
```

Each stage is a standalone Bun/TypeScript CLI. Each stage's output is the next stage's input. All data flows through JSONL files.

## How to Run the Pipeline

**Prerequisites:** Bun runtime, `ANTHROPIC_API_KEY` env var (for M3 and M4).

```bash
cd plugins/skill-eval/scripts
bun install  # one-time, installs @anthropic-ai/sdk

# M1: Extract events from session transcripts
bun transcript-extract.ts --since 7d --output ../events.jsonl

# M2: Classify events into friction/success/noise
bun signal-classify.ts --input ../events.jsonl --output ../classified-all.jsonl

# M3: Identify optimization targets via LLM-as-judge
bun target-identify.ts --input ../classified-all.jsonl --output ../targets.jsonl

# M4: Run autoresearch loop on top target
bun autoresearch-loop.ts --targets ../targets.jsonl --target-rank 1
```

Each script supports `--help` for full option listing.

## Key Decisions Made (and Why)

### 1. Rule-based classification (M2) over LLM-assisted
We chose deterministic regex/heuristic classifiers over LLM calls for M2. Reasons:
- Cost: M2 processes thousands of events; LLM calls per-event would be expensive
- Determinism: same input always produces same classification
- Speed: sub-second for 10k+ events
- The 14 friction detectors cover the patterns that actually matter in practice

The tradeoff: novel friction patterns that don't match existing heuristics will be classified as "neutral." If this becomes a problem, an LLM fallback classifier can be added as a second pass.

### 2. Binary yes/no eval questions (M3) over quantitative thresholds
Early M3 generated eval questions like "Does the fix reduce error rate by ≥80%?" These are unverifiable from a patch alone — you'd need runtime data. We switched to structural questions like "Does the patch include error handling for timeout cases?" which a judge can answer from the patch text.

This was the single most impactful change. M4's perfect scores came directly from this fix.

### 3. Append-only JSONL experiment logs (M4) over git branching
The Karpathy autoresearch pattern uses `git commit` on keep and `git reset --hard` on discard. We opted for JSONL logs instead because:
- Simpler to implement and debug
- Full history preserved (including failed attempts, which are informative)
- No git state pollution during experimentation
- Git-based keep/revert is a future enhancement, not a blocker

### 4. String-aware JSON parser over regex extraction
Haiku returns JSON with code patches containing `{` and `}` in string values. A naive balanced-brace parser breaks. The `extractJson()` function tracks whether it's inside a JSON string literal (including escape sequences) before counting brace depth. This was a critical fix — without it, ~50% of M4 runs crashed.

### 5. System message for JSON enforcement
Adding `"You are a JSON-only responder. Always respond with a single valid JSON object."` as a system message eliminated Haiku's tendency to wrap JSON in conversational prose. Without it, roughly 1 in 3 judge calls returned unparseable responses.

## What's Done

- **M1**: Complete. Parses user messages, assistant text, tool_use, tool_result, thinking blocks, subagent transcripts. Handles compaction boundaries and system messages.
- **M2**: Complete. 14 friction detectors, noise filtering, confidence scores. Rule-based, deterministic.
- **M3**: Complete. Clusters friction by subcategory+tool_name, scores by frequency×log2(session_count)×severity×improvability. Generates 3-6 binary eval questions per target.
- **M4**: Complete. Generate→eval→keep/discard loop with plateau detection (5 discards), crash bailout (3 crashes), and perfect-score early stop.
- **Plugin packaging**: Complete. All scripts have --help, SKILL.md docs updated, registered in marketplace.json.

## What's NOT Done (Remaining Work)

### Must-Do

1. **Manual validation passes (M2, M3)**
   - Review 50 classified events for precision/recall of friction detection
   - Verify no true friction signals are classified as noise
   - Review 10 identified targets — do they match intuition?
   - These are checkbox items in MILESTONES.md under M2/M3 Validate sections

2. **Standalone eval-harness.ts (M4)**
   - The eval logic is currently inline in `autoresearch-loop.ts`
   - Extract into `scripts/eval-harness.ts` so it can be run independently
   - Use case: re-evaluate a previous experiment's artifacts without re-running the full loop

3. **Git-based keep/revert (M4)**
   - Currently M4 only logs results to JSONL — it doesn't actually commit improvements to git
   - The design doc (`design/M4-autoresearch-loop.md`) describes the intended flow
   - Implementation: `git commit` on score improvement, `git reset --hard` on regression
   - This turns M4 from "advisory" to "actually modifies artifacts"

### Should-Do

4. **Score gaming detection (M4)**
   - Verify that high-scoring patches actually improve the artifact, not just satisfy eval questions trivially
   - Could be a post-hoc review step or a secondary judge with different questions

5. **Telemetry hooks (cross-cutting)**
   - Optional: capture pipeline run metadata (runtime, event counts, target scores) via Claude Code hooks
   - Low priority — the JSONL logs already capture most of this

6. **CASS integration path (M1)**
   - M1 currently parses raw transcripts from `~/.claude/projects/`
   - An alternative input path via CASS CLI export was designed but not implemented
   - Only needed if CASS becomes the primary session storage

## Risk Areas

### 1. Haiku reliability for code generation
M4 uses Haiku for both generation and evaluation. Haiku is cost-effective but occasionally produces malformed output even with the system message fix. On complex targets (large code patches), expect ~10-20% crash rate per round. The 3-crash bailout prevents infinite loops, but switching to Sonnet for generation (while keeping Haiku for eval) would improve reliability at ~5x cost.

### 2. Eval question quality is the ceiling
The entire M4 loop is only as good as M3's eval questions. If the questions are vague, compound, or unverifiable from the patch text, the loop will plateau or produce false positives. When adding new target types, invest time in the eval question design.

### 3. Transcript schema changes
M1 parses Claude Code's internal JSONL format, which is not a public API. If Anthropic changes the transcript schema (message types, content block structure, tool_use format), M1 will break silently — it'll just produce fewer events. The `types.ts` file documents the current schema; check against real transcripts periodically.

### 4. Classification blind spots (M2)
The 14 friction detectors cover known patterns. Novel failure modes (e.g., a new type of tool error, or a subtle user frustration pattern not caught by regex) will be classified as "neutral" and invisible to M3. Periodically sample "neutral" events to check for emerging friction patterns.

### 5. No end-to-end integration test
Each milestone has been tested individually and the pipeline has been run end-to-end manually. There is no automated test that runs M1→M2→M3→M4 on a fixture dataset and asserts expected outputs. This would be valuable for regression detection.

## Architecture Notes for the Next Agent

- **Type system**: All shared interfaces are in `scripts/types.ts`, including event types, classification types, target/experiment types. The discriminated unions (`ExtractedEvent`, `RawContentBlock`) use a `type` field. Follow this pattern for any new event types.
- **JSONL everywhere**: Every stage reads and writes newline-delimited JSON. One record per line. This is deliberate — it enables streaming, `wc -l` for counts, `head`/`tail` for sampling, and `jq` for ad-hoc queries.
- **Anthropic SDK**: M3 and M4 use `@anthropic-ai/sdk` directly (not via Claude Code's /zo/ask or --print). This is simpler, cheaper, and gives full control over system messages and temperature.
- **The `extractJson()` function** lives in `scripts/shared.ts` and is imported by both M3 and M4.
- **Model selection**: M3 and M4 default to `claude-haiku-4-5-20251001` via `--model` flag. The model string is passed directly to the Anthropic SDK. Any model available on the API works.
